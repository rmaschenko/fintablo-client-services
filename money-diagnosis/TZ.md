# ТЗ v2: Микросервис «Диагностика финансового учёта»

**Приоритет запуска:** 1 (первый).
**Владелец:** Роман (маркетинг + PR, Финтабло).
**Целевой URL:** fintablo.ru/tools/diag/financial/
**Канал:** Яндекс.Директ (весь трафик — проектный и непроектный).

## Цель сервиса
Генерировать тёплые B2B-лиды, прогретые **реальным AHA-моментом** (не синтетической цифрой), с высокой контактностью и ожиданием звонка. Прогрев на покупку Финтабло идёт через: (1) звонок менеджера с разбором, (2) 7-письменную PLG-серию, (3) ценность отчёта.

## ICP и отсечка
Мягкий фильтр: **только** по обороту `< 2 млн ₽/мес` → альтернативный экран (фриум-регистрация + кейс + Telegram), в CRM не отправляем. Всех остальных (любая бизнес-модель, любая отрасль) пускаем в основную воронку — непроектных не сегментируем.

## Сбор данных: 5 шагов квиза

**STEP_ORDER:** `[0, 1, 2, 3, 4, 5, 65, 10]`

- **0** — Стартовый экран (H1, social proof «2 200+ компаний», 3 буллета, CTA «Начать диагностику»)
- **1** — **Роль** (собственник / финансовый директор / финансист / другое)
- **2** — **Оборот** (слайдер с дискретными зонами 500к–100 млн+/мес)
- **3** — **Отрасль** (6 карточек: стройка, IT, агентство, производство, услуги, другое) — **для персонализации формулировок**, не фильтр
- **4** — **Система учёта** (нет / Excel / 1С / другой сервис / специализированный сервис) — single choice (не multi, упрощение)
- **5** — **Главная боль** (single choice из 5) — **драйвер AHA-экрана**:
  - `margin_blind` — Не вижу маржу по направлениям
  - `late_loss` — Узнаю об убытках задним числом
  - `cash_surprise` — Не понимаю, когда будет кассовый разрыв
  - `data_lag` — Цифры приходят с задержкой, решения — наугад
  - `no_big_picture` — Учёт есть, картины нет
- **65** — **AHA-экран** (один сфокусированный, 3-слойный — см. `AHA-MATRIX.md`)
- **10** — **Гейт контакта** (двойной: телефон + email; имя менеджера/фото; без выбора слота)

После шага 10 — редирект на `report.html`. Больше вопросов нет.

## Прогресс-бар
Одна линия «Шаг N из 5». Без деления на блоки — так короче воспринимается.

## AHA-экран (шаг 65) — детали
См. `AHA-MATRIX.md`. Кратко:
- Имя профиля (6 универсальных типов) + 2 строки описания
- 3 зеркальных вопроса (из выбранной боли, с подстановкой `{unit}` по отрасли)
- 1 бенчмарк (отраслевой + модификатор по обороту)
- CTA «Получить подробный разбор»

**Никакого дашборда из 3 карточек.** Цифра `estimatedAnnualLoss` уходит в отчёт, не в AHA.

## Расчётная модель
- `monthlyRevenue`, `annualRevenue` — из шага 2
- `riskCoeff` — по системе учёта (none=0.20, excel=0.13, 1c=0.10, other=0.08, service=0.04)
- `estimatedAnnualLoss = annualRevenue × riskCoeff` — в отчёт, не в AHA
- `transparencyIndex` (0–100) — базовый балл системы ± модификаторы — в отчёт
- `profileType` — 6 типов, классификация по `revenue + system + industry + primaryPain`
- `icpScore` (0–100) → `lead_A` (61+), `lead_B` (31–60), `lead_C` (0–30)

## Отчёт (`report.html`) — одностраничный

5 секций на одной странице, без блюра:
1. **Профиль** (имя + 2 строки) + ответы на 3 зеркальных вопроса AHA
2. **Оценка потерь** (`estimatedAnnualLoss` + `transparencyIndex`) — с хеджем «по нашим данным»
3. **Где теряется** (3 карточки по отрасли + боли)
4. **Как Финтабло закрывает ваш профиль** (3–5 функций под профиль + 1 кейс)
5. **Подтверждение заявки** (имя менеджера, фото, что он сделает, CTA «открыть Финтабло»)

## Страница благодарности (`thankyou.html`)
- Персональный заголовок с именем
- 3 шага «что дальше» (без конкретного времени — SLA нет)
- Динамический блок с результатами диагностики (профиль, главная цифра)
- Имя/фото менеджера
- Telegram-канал «Поставлено» — soft-prize
- Кейс + регистрация на фриум Финтабло

## Гейт контакта (шаг 10)
- **Телефон** — обязателен, маска `+7 (999) 999-99-99`
- **Email** — обязателен, базовая валидация
- Honeypot-поле `website`
- Имя менеджера + фото + текст «Менеджер свяжется в рабочее время с разбором»
- CTA «Открыть отчёт»
- Без выбора слота

## Фирменный стиль
Inter 4 начертания (400/500/600/700, локально, 152-ФЗ). Токены брендовых цветов и радиусы — из текущего `css/style.css`. Mobile-first, iPhone SE 375×667 — минимум.

## Яндекс.Метрика (префикс `moneydiag_`)
`landing_view`, `start`, `role_selected`, `revenue_selected`, `industry_selected`, `system_selected`, `pain_selected`, `aha_view`, `contact_form_open`, `lead_submitted`, `report_view`, `thankyou_view`, `abandoned_after_aha`

**Сегменты пикселя:**
- **hot** — дошёл до AHA + провёл на нём > 10 сек + дошёл до отчёта
- **warm** — начал квиз, но не дошёл до гейта
- **cold** — зашёл и ушёл с landing

**Запрещено (из редполитики):** гарантии результата, эмодзи в H1, восклицательные знаки, «ты», «эффективный», «удобный», «инновационный», SaaS-штампы («в одном окне», «в пару кликов»), превосходные степени без доказательств, смета/КС-2/Гант/таймшиты/склад.

## Что изменилось по сравнению с v1
- Было: 10 шагов с AHA-дашбордом из 3 карточек (синтетические цифры) → Стало: 5 шагов + AHA-текст с профилем/зеркальными вопросами/бенчмарком.
- Было: раздельные отчёты по ролям → Стало: единый отчёт с адаптацией лексики.
- Было: `activeProjects` обязателен → Стало: убран (не нужен для универсального AHA).
- Было: `mainProblems` мультивыбор + `desiredResult` + `hasFinancist` → Стало: одна `primaryPain` single-choice.
- Было: выбор слота перезвона → Стало: нет SLA, просто имя менеджера.
- Было: universe проектного бизнеса → Стало: любой бизнес (непроектных не отсекаем).
- Сохранено: DS Kit, шрифты, токены, 152-ФЗ, AmoCRM-интеграция, CSV-бэкап, 3 сегмента пикселя.
