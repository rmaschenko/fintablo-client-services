# Шаблон: создание нового клиентского микросервиса Финтабло

> **Кому:** новому диалогу Claude Code, который берётся за свежий PLG-микросервис Финтабло.
> **Что делать:** прочитать это **полностью** перед первой строкой кода. Тут все общие правила, которые не нужно переспрашивать. Специфичное ТЗ на конкретный сервис придёт отдельным файлом.

---

## 0. Контекст

У нас семья клиентских PLG-микросервисов Финтабло под платный трафик Яндекс.Директ. Каждый — статический лендинг+квиз+калькулятор+форма лида, отдающий контакты в AmoCRM. Цель каждого — перехватить ICP в момент конкретной боли и через персонализированный AHA показать ценность Финтабло, собрав контакт для звонка эксперта.

Эталонный работающий пример — `money-profit/` в этом же репозитории. **Смотри его как референс** по любым неуказанным вопросам.

## 3. Фирстиль и шрифты

**Источник истины:** growth_cfo (`~/Вайбкод/fintablo/`) — там же, что и основной сайт fintablo.ru. Это ключевое условие для message match трафика из Яндекс.Директа.

### Шрифты (все локально, только Inter):
- **Inter** 400/500/600/700 — единственный шрифт для всего UI: body, заголовки, числа.
- Числа в финансовых таблицах и KPI — тот же Inter с `font-variant-numeric: tabular-nums lining-nums`.

**Не использовать:** IBM Plex Sans/Mono, Literata, любые serif-шрифты. Был короткий период Design System Kit с другими шрифтами — отклонён пользователем.

## 5. UX-принципы PLG-сервиса (из growth_cfo + money-profit)

### 5.1 Иллюзия прогресса
Прогресс-бар показывай с **первого шага квиза**, не с середины. Даже 10% заполненности мозг воспринимает как обязательство. Используй именованные стадии в хедере прогресса («О бизнесе › Ваши цифры › Получить анализ»).

### 5.2 Один вопрос на экран
Никакого скролла, никакой перегрузки. Макс 12 слов в вопросе. Варианты — 2–5 карточек, не больше (парадокс выбора Шварца). Карточка со вторым кликом **не нужна** — после клика auto-advance через 1.2 сек + пауза для insight-текста.

### 5.3 Микро-награды после каждого выбора
После каждого ответа — `insight-box` с короткой подстройкой под ответ (например: «Так отвечают 67% собственников с оборотом 5+ млн. Они обычно упираются в Х»). Это не украшение — это поддержание дофаминовой петли.

### 5.4 Плавающая CTA внизу (sticky-cta)
Появляется **только** когда нужная форма валидна (например, все поля калькулятора заполнены). Слайд снизу с transform+transition .3s. Не перекрывает контент (`body:has(.sticky-cta.visible) footer { padding-bottom: 140px }`).

### 5.5 Кнопка «Далее» disabled до валидного состояния
Никакого серого текста «заполните поле». Кнопка просто не активна — это ускоряет понимание требования без лишних слов.

### 5.6 Кнопка «Назад» всегда доступна (кроме шага 1)
Фиксирована в `.step-nav` внизу, первая кнопка слева.

### 5.7 Anti-ICP-фильтр
Если в ТЗ описан Anti-ICP (обычно по обороту/размеру бизнеса) — мягкое отсечение экраном с предложением другого полезного действия (trial / блог), **не** отправлять таких в CRM. Оставь кнопку «Всё равно продолжить» на случай ошибки выбора.

### 5.8 Персональный разбор **перед** формой контакта (PLG-гейт)
Ключевой паттерн: после калькулятора покажи **персонализированный разбор** (аха-момент) — и **только потом** форму контакта. В разборе должны быть:
- severity-пилл + PAS-заголовок
- визуальный график с подсветкой проблемы
- 3 KPI-карточки
- **«3 типовые точки утечки в вашей отрасли»** с помёткой «🔒 в [Финтабло]» — это и есть гейт: пользователь видит КАКИЕ бывают утечки, но КАКОЙ ИМЕННО его проект — только в Финтабло
- PLG-блок «что даёт Финтабло» с 4 функциями

Это воплощение принципа из ТЗ: «первый слой бесплатно, второй слой — внутри продукта. Незавершённое действие = мотивация купить».

### 5.9 Форма лида максимум 3 поля
Имя (обяз.) + телефон (обяз., маска `+7 (XXX) XXX-XX-XX`, валидация операторских кодов + антифейк) + email (опционально). Честный consent-текст, без чекбоксов (согласие через факт нажатия кнопки). Honeypot-поле `<input name="website" class="hp-field">`.

### 5.10 Эффект Зейгарник (beforeunload)
На шагах, где есть введённые данные — предупреждение при попытке закрыть вкладку. Отключать перед редиректом на thankyou: `window._<service>_navigating = true`.

### 5.11 State TTL 48 часов
Сохраняй прогресс в `localStorage` с ключом `<service>_state_v1` и полем `_ts`. При возврате — баннер «У вас есть незавершённый расчёт · Продолжить / Начать заново».

### 5.12 UTM first-click attribution
Захватывай UTM в `sessionStorage` при первом заходе, **не перезаписывай** при повторном.

## 6. Tone of voice

**Обязательно:**
- Серьёзный CFO-тон. Термины профессиональные: ДДС, ОПиУ, кассовый vs начислительный, дебиторка, маржа, LTV/CAC.
- Хеджирующие формулировки: «по введённым данным», «как правило», «типичная ситуация», «около X ₽», «потенциальный разрыв».
- Русский язык: «Финтабло» (не FinTablo), «—» тире (не дефис), «» ёлочки (не "").
- Числа по ГОСТ 8.417-2002: `1 234 567,89 ₽` (пробел-разделитель, запятая-десятичная).

**Запрещено:**
- Emoji в заголовках и body (кроме иконок карточек 🏗️💻 и системных маркеров 📌🔴✅).
- Игривые/разговорные фразы.
- Утверждения-гарантии: «вы теряете», «мы гарантируем», «ваш бизнес убыточен».
- Восклицательные знаки в заголовках, «!!!», КАПС.
- Геймификация ради геймификации.

**Никогда не упоминай в контексте Финтабло:**
- Сметы / КС-2 / КС-3
- Диаграмма Ганта / план-график
- Таймшиты / учёт рабочего времени

**Зона силы продукта (упоминай):**
- Платёжный календарь
- Прогноз ДДС
- P&L по проектам
- Бюджет и план-факт
- Контроль дебиторки

## 7. Обязательные события Яндекс.Метрики

Для каждого сервиса минимум:
- `<prefix>_landing_view` — открытие страницы
- `<prefix>_start` — клик главной CTA на welcome
- `<prefix>_step_N` — каждый шаг квиза
- `<prefix>_data_entered` — заполнены все поля калькулятора
- `<prefix>_gap_detected` / `<prefix>_diagnosis_<type>` — аналитическая цель
- `<prefix>_report_view` — пользователь увидел персональный разбор
- `<prefix>_contact` — открыл форму контакта
- `<prefix>_lead` — **главная цель: успешная отправка формы**
- `<prefix>_thankyou` — открытие thankyou.html
- `<prefix>_share` — клик «Поделиться»
- `<prefix>_abandoned_50+` — beforeunload на середине

Передавай параметры: `diagnosisType`, `industry`, `revenue_range` и т.п.

## 8. AmoCRM интеграция

### Шаблон `api/lead.php` (скопировать из `money-profit/api/lead.php`)
Поддерживает:
1. **CSV-backup всегда** — `api/leads/<service>_YYYY-MM.csv` (UTF-8 BOM, `;` разделитель, `.htaccess Deny from all`). Работает даже если AmoCRM выключен.
2. **AmoCRM `/api/v4/leads/complex`** — опционально, активируется если в `../.env` заполнены:
   ```
   AMO_DOMAIN=tablo2.amocrm.ru
   AMO_TOKEN=eyJ0eXAi…
   AMO_PIPELINE_ID=5278171
   AMO_STATUS_ID=47065159
   ```
   `.env` **в gitignore**, конечные токены даст владелец.
3. **Honeypot:** если пришло поле `website` — тихо возвращаем 200 и не обрабатываем.
4. **Теги AmoCRM (обязательно):**
   - базовые: `micro_service`, `<имя_сервиса>`, `direct_client`
   - квалификация: `lead_A` (критичный) / `lead_B` (нормальный) / `lead_C` (anti-ICP)
5. **Название лида:** `<Тема сервиса>: <Имя> — <отрасль> — <диапазон>`
6. **Заметка к лиду:** весь расчёт + UTM + метаданные в читаемом виде.

## 9. Технические требования

- **Только статический HTML/CSS/JS**, без сборщиков и фреймворков
- **PHP 8.0+** на бэке (api/lead.php)
- **Mobile-first**, iPhone SE (375×667) — минимальный таргет
- **Touch-targets ≥ 48px** для кнопок
- **Debounce 300–400мс** для live-расчётов
- **Анимации переходов 200–250мс**, `cubic-bezier(.4,0,.2,1)`
- **Прелоад критичных шрифтов** (`<link rel="preload" href="fonts/..." as="font" crossorigin>`) — минимум H1 и body 400
- **Favicon:** `https://fintablo.ru/favicon.ico` (не свой)
- **OG-теги** для соц-шаринга обязательны
- **viewport meta:** `width=device-width,initial-scale=1,viewport-fit=cover`
- **theme-color:** `#1A1F36` (navy)
- **Все расчёты в JS локально** — никаких API-вызовов в логике калькулятора. API только для `api/lead.php`.

## 10. Процесс работы с пользователем

1. **Спроси конкретное ТЗ** на сервис (обычно приходит файлом `TZ.md`). Не начинай без него.
2. **Прочитай ТЗ на 100%** перед кодом. Если ТЗ длинное — используй Explore-агента для экстракции.
3. **Сравни с `money-profit/`** — какие компоненты уже готовы к копированию, какие надо писать с нуля.
4. **Делай коммиты по логическим кускам**. Не мегакоммит в конце. Типичная последовательность:
   - `chore: scaffold <service>`
   - `feat: welcome + quiz`
   - `feat: calculator + viz`
   - `feat: personal report`
   - `feat: lead form + API`
   - `feat: thankyou + cookie`
5. **После каждого коммита — push** и дай юзеру ссылки:
   - `http://localhost:8765/` (открыть в браузере)
   - ссылка на коммит в GitHub
6. **НЕ МЕНЯЙ fintablo-client-services/money-profit/** без явной просьбы. Это эталон, его ломать нельзя.
7. **Не трогай FTP** (см. §2). Не пытайся «задеплоить». Не создавай деплой-скрипты.
8. **Давай кликабельные ссылки** вместо терминальных команд. Юзер — не ИТ-специалист.
